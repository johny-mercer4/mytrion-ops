You are designing the **Retention Mytrion** ‚Äî one department module inside Octane's internal
CRM (React + Vite SPA, dark-first with a light theme). Octane sells fuel cards to trucking
carriers; the Retention department wins back carriers who stopped fueling. Today the module

design, not a concept board.

## 1. How Retention works (the domain in 60 seconds)

A nightly sync scans the data warehouse and **auto-creates a case** for every active,
non-debtor carrier whose card activity breached its expected cadence:

- Every carrier gets a **frequency class** from its last 90 days: `high` (transaction
  expected every 2 days), `medium` (every 5), `low` (every 7).
- If days-since-last-transaction exceeds the threshold, a case is born. If the client fuels
  again, the sync **auto-closes the case as `returned`** ‚Äî reps never see it again.

Each open case sits on a **phase ladder** (who owns the outreach right now):

| Phase | Meaning |
|---|---|
| `sales` | The deal's own sales agent gets the first window (2 business days) to re-engage |
| `retention` | The Retention desk works the case ‚Äî calls, reason discovery, offers |
| `open_pool` | Released: other agents can request to take ownership |
| `citi` | **Final.** No further outreach; never auto-closed; end of the ladder |

Within a phase, a case has a **stage** (how the rep works it): `inactive_no_reason`
(priority ‚Äî call first), `inactive_reason_noted` (low-frequency check-ins),
`out_of_reach` (client not picking up; max 5 attempts, tracked per case), `pending`
(re-engagement underway), `assigned_to_agent` (hands off unless the agent stalls).

Cases close with an **outcome**: `returned` / `saved` (good) ¬∑ `refused_offer` /
`out_of_business` / `no_response` / `lost` (bad).

## 2. Data contract (live API ‚Äî design against these exact fields)

Typed client already exists at `src/api/retention.ts`. Endpoints:

- `listRetentionCases({ phase?, status?, stage?, carrierId?, limit, offset })` ‚Üí `{ cases, total }`
- `getRetentionCase(id)` / `createRetentionCase(input)` / `updateRetentionCase(id, patch)`
- `deleteRetentionCase(id)` ‚Äî **admin-only**
- `runRetentionSync({ lookbackDays?, limit? })` ‚Äî **admin-only**, returns
  `{ summary: { scanned, breached, created, refreshed, closedReturned } }`

`RetentionCase` (everything the UI can show):

```ts
{
  id: string;                        // 'rc_‚Ä¶'
  carrierId: string;                 // e.g. '5796646'
  companyName: string | null;        // 'HORSERIDER INC'
  applicationId: string | null;
  agentName: string | null;          // deal owner (sales agent)
  agentZohoUserId: string | null;
  phase: 'sales' | 'retention' | 'open_pool' | 'citi';
  phaseChangedAt: string;            // ISO ‚Äî drives "N days in phase"
  stage: 'inactive_no_reason' | 'inactive_reason_noted' | 'out_of_reach'
       | 'pending' | 'assigned_to_agent';
  status: 'open' | 'closed';
  outcome: 'returned' | 'saved' | 'refused_offer' | 'out_of_business'
         | 'no_response' | 'lost' | null;
  closedAt: string | null;
  inactivityReason: string | null;   // free slug: 'vacation', 'low_discounts', ‚Ä¶
  reasonNote: string | null;         // rep's free text
  outOfReachAttempts: number;        // 0‚Äì5; at 5 the case should move to open_pool
  frequencyClass: 'high' | 'medium' | 'low' | null;
  thresholdDays: number | null;      // 2 / 5 / 7
  lastTransactionAt: string | null;
  daysInactive: number | null;       // THE headline number, read against thresholdDays
  txCount90d: number | null;
  gallons90d: number | null;         // volume = client value; sync orders by this
  activeCards: number | null;
  poolAssignment: 'available' | 'requested' | 'assigned' | 'rejected' | null; // open_pool only
  poolTakenBy: string | null;
  source: 'auto' | 'manual';         // auto = generated by the nightly scan
  lastSyncedAt: string | null;
  createdAt: string; updatedAt: string;
}
```

RBAC the UI must respect: reads and case-work writes need the `retention` department
(sent as the `x-department-access` header ‚Äî transport handles it); **delete and sync are
admin-only** (hide or disable for non-admins). A 403 means the signed-in worker lacks the
department ‚Äî design that state, don't let it look like an error.

`apps/mytrion-crm/src/mytrions/retention/` ‚Äî `index.tsx` renders `MytrionShell` (shared

## 4. Design tasks

1. **Cases board, re-founded on the real model.** Recommended: kanban columns = the four
   phases (Sales ‚Üí Retention ‚Üí Open Pool ‚Üí Citi), stage shown as a chip on the card ‚Äî
   but if you find a stronger structure (phase rail + stage lanes, split board + queue),
   argue for it. Must answer at a glance: *who should I call first?* Priority signal =
   `inactive_no_reason` + high `gallons90d` + `daysInactive` far past `thresholdDays`.
2. **Case card.** Company + carrier id, days inactive vs threshold (e.g. "9d quiet / 2d
   cadence" ‚Äî make the breach severity legible), frequency-class badge, 90-day gallons,
   agent, stage chip, reason if noted, out-of-reach pips (n of 5), `auto`/`manual` origin
   mark. Dense but scannable; these boards will hold 50‚Äì200 cases.
3. **Case detail** (drawer or modal, evolve `CaseDetail.tsx`): full field readout + the
   actions ‚Äî move phase (forward emphasis, Citi guarded by a confirm since it's final),
   set stage, pick/edit inactivity reason + note, bump out-of-reach attempts, close with
   outcome (good/bad outcomes visually distinct), and admin-only delete.
4. **Open Pool tab = cases where `phase === 'open_pool'`.** Table or card list keyed on
   `poolAssignment` with the request ‚Üí assign/reject flow (`poolTakenBy` on assignment).
   Show `daysInactive`, gallons, reason ‚Äî what a claiming agent needs to decide.
5. **Closed cases.** A `status: 'closed'` view (filter or segment) grouped by outcome ‚Äî
   the team's save-rate at a glance. Auto-closed `returned` cases should read as wins.
6. **Sync surface (admin).** "Sync now" affordance + last-synced time + result toast
   ("Scanned 500 ¬∑ 12 new ¬∑ 3 returned"). Non-admins see the freshness, not the button.
7. **States.** Loading (skeletons), empty ("no open cases ‚Äî the nightly scan runs at
   05:00"), error/retry, 403 department-gate, and optimistic updates for drag/patch with
   rollback toast.

## 5. Constraints

- Stay inside `MytrionShell`; CSS modules; `lucide-react` only ‚Äî no new dependencies,
  no component library.
- Theme via existing CSS custom properties (`--accent`, `--danger`, `--warning`,
  `--success`, `--purple`, glass/`--bg` layers). Retention's hue is **danger/red** in the
  Mytrion picker ‚Äî use it as the module accent, not as an alarm on every element.
- Both themes, dark first. WCAG AA contrast. Full keyboard support on the board
  (kanban drag must have a menu/button fallback).
- Language: plain and operational ("9 days quiet", "Take this case"), not corporate.
